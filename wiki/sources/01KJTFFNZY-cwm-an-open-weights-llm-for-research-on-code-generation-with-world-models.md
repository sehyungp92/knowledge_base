# CWM: An Open-Weights LLM for Research on Code Generation with World Models

CWM (Code World Model) is a 32B-parameter open-weights LLM from FAIR's CodeGen team that introduces a dedicated mid-training stage on execution trace data and large-scale agentic trajectories, with the thesis that mastering code requires understanding what programs *do* when executed — not just what they *look like*. The paper presents early evidence that world-model-grounded representations improve downstream SWE-bench performance, and opens this research direction to the community via weights released under a noncommercial license.

**Authors:** FAIR CodeGen team (Jade Copet, Quentin Carbonneaux, Gal Cohen, Jonas Gehring, et al. — 51 total)
**Published:** 2025-09-30
**Type:** Paper
**Source:** https://arxiv.org/pdf/2510.02387

---

## Motivation

Standard LLM pre-training treats code as text — models learn syntax and statistical co-occurrence patterns but are never explicitly taught what code *does* when executed. The authors argue this ceiling is already visible: benchmark after benchmark reveals shortcomings on every frontier model release, even as raw performance climbs. Skills instrumentally important to software engineers — tracking how a line changes variable state, predicting how a codebase change affects outputs, reasoning through a failing test — are not naturally acquired from next-token prediction on source text alone.

A secondary motivation is access: no open-weights model of comparable scale existed as a testbed for world-model-driven code generation research, concentrating this direction in proprietary labs.

---

## Architecture and Training

CWM is a dense decoder-only LLM trained in three stages:

1. **General pre-training** — 8T tokens, 8k context
2. **Code world model mid-training** — 5T tokens, 131k context (the novel contribution)
3. **Post-training** — SFT + RL

The 131k context mid-training eliminates the need for a separate long-context fine-tuning phase common in other recipes.

### Mid-Training Data: Execution Traces

The first CWM-specific dataset is Python execution traces — over **120M function-level traces** capturing stack frame state (local variables as JSON dicts) after each executed line, formatted as observation-action trajectories with custom separator tokens. Coverage includes:

- **Standalone functions** traced from open-source Python code
- **Repository-level tracing** across 21,000+ executable repo images, yielding ~70,000 execution-traced commits, built using the novel *Activ* pipeline that repurposes GitHub Actions CI infrastructure
- **CodeContests solutions** for competitive programming coverage
- **Natural language traces** — 75M trajectories generated by Qwen3-32B-FP8 re-expressing trace content in natural language, enabling knowledge transfer to reasoning contexts

Traces exceeding 10,000 line events or 1 MB are filtered out, introducing a bias toward short, simple functions that is a known limitation.

### Mid-Training Data: ForagerAgent

The second CWM-specific dataset is **ForagerAgent** — 3M multi-step agentic trajectories generated by prompting Llama3-70B-Instruct or Qwen3-235B-A22B to fix bugs (*mutate-fix*) or resolve real GitHub issues (*issue-fix*) in Docker-sandboxed repository environments. Key design choices:

- Included at **mid-training scale** rather than post-training fine-tuning, the hypothesis being that it shapes internal representations before RL rather than fine-tuning surface behavior
- The model is trained to predict **both agent and environment turns** (with 50% stochastic loss masking on observations) to learn environment dynamics, not just policy
- Trajectories are **not filtered by task success** — the model sees both successful and failed bug-fix attempts, a deliberate choice to build a comprehensive world model of agentic interactions

### Post-Training

SFT runs on 100B tokens including iteratively self-bootstrapped SWE reasoning traces. Joint RL then runs across four environments simultaneously: agentic SWE, competitive coding, agentic coding, and mathematics.

RL uses a custom multi-turn asynchronous GRPO variant with several modifications over standard GRPO:
- No σ normalization (replaced with centered return)
- No length normalization (divided by max context size instead)
- Clip-higher (ε_high = 0.25)
- Gibberish detection via rare-token filtering
- Length reward schedule that penalizes overlong correct solutions early in training, then phases out

The SWE RL environment uses a minimal four-tool design (bash, edit, create, submit) with a hybrid reward: +1 for test-passing patches, 0 for similarity ≥ 0.5, −1 otherwise. Reasoning tokens from SFT are replaced with plain `<think></think>` tags at RL time to increase exploration entropy.

## Results

| Benchmark | Score |
|---|---|
| SWE-bench Verified (pass@1, test-time scaling) | **65.8%** |
| SWE-bench Verified (base, no scaling) | 53.9% |
| SWE-bench Verified (majority vote k=24) | 58.4% |
| SWE-bench Verified (pass@40) | 80.4% |
| LiveCodeBench-v5 | **68.6%** |
| Math-500 | **96.6%** |
| AIME 2024 | **76.0%** |
| CruxEval-Out | **94.3%** |

CWM outperforms open-weight models at comparable size and is competitive with larger proprietary models including GPT-oss-120B on SWE-bench. The pass@40 of 80.4% confirms strong sample diversity and high ceiling for test-time compute investment.

Self-bootstrapping SWE RL traces were important to the training recipe: the SFT checkpoint started near 0% on SWE-bench (format errors), and iterative bootstrapping raised it to 30% → 37% → 43% across three rounds before joint RL.

## Capabilities

### Neural Code Interpretation
CWM can simulate Python program execution step-by-step — predicting interpreter stack frames, local variable states, and return values at each executed line. This is described as an early-stage capability (maturity: **demo**), with the authors explicitly positioning formal integration into reasoning chains as future work.

### Execution-Grounded Program Synthesis
Under appropriate prompting, CWM can predict execution traces for functions *not given in context* — constrained only by natural language descriptions or formal assert statements — then construct function bodies that produce consistent traces. This reverses the usual direction of trace prediction (maturity: **demo**).

### Agentic SWE with Test-Time Scaling
CWM constructs input-output pairs to self-verify code correctness during competitive programming, without being explicitly trained for this specific behavior — an emergent capability attributed to world-model grounding (maturity: **demo**).

### Environment-Grounded Representations from Mid-Training
Mid-training on 3M ForagerAgent trajectories at scale establishes environment-grounded representations before post-training RL, with the authors arguing this improves downstream training efficiency compared to post-training-only agentic data (maturity: **research_only**).

## Limitations and Open Questions

### Fundamental Gaps in Execution Modeling

The execution trace approach has hard scope limits baked into the training data:

- **Global variables and external side effects are excluded** — only local variable states are modeled, severely limiting fidelity for programs with I/O, shared state, or external API calls (severity: significant, trajectory: stable)
- **Long-running programs are excluded** — traces exceeding 10,000 line events or 1 MB are filtered, biasing the model toward short, simple functions (severity: significant, trajectory: stable)
- **Python-only** — no evidence of support for JavaScript, TypeScript, C++, Java, or other languages used in production software engineering (severity: significant, trajectory: unclear)

### World Modeling Benefits Remain Preliminary

> *"We present first steps of how world models can benefit agentic coding... However, we believe the best is yet to come."*

The paper does not provide ablations isolating the causal contribution of execution trace data to downstream performance. World modeling benefits are positioned as an "early results" finding, not a rigorously demonstrated mechanism (severity: significant, trajectory: improving).

Formal integration of trace prediction into natural language reasoning chains is prototype-only — the capability emerges under light prompting but was not trained for explicitly, and no principled pipeline exists yet (severity: significant, trajectory: improving).

### Infrastructure Scaling Challenges

Repository-level execution tracing requires building executable repository images at scale — automated LLM-backed setup fails for a substantial fraction of real-world codebases, and historical commits fail at even higher rates due to dependency version drift (severity: minor, trajectory: stable). This creates sampling bias toward recent, well-maintained codebases.

### Deployment Constraints

- **Hardware requirements**: Even quantized, CWM requires a single 80 GB H100 — excluding consumer hardware, CPU inference, and most edge deployments (severity: significant, trajectory: improving)
- **Noncommercial license**: Commercial deployment is prohibited, blocking production SWE adoption and limiting real-world validation (severity: significant, trajectory: unclear)
- **Test-time scaling dependency**: The headline 65.8% score requires parallel candidate generation and patch ranking — single-attempt base performance is 53.9%, and inference cost scales proportionally to candidate count (severity: workaround_exists, trajectory: improving)

### Training Data Quality Bounds

- Natural language trace quality is bounded by Qwen3-32B generation fidelity — a non-trivial proportion of traces were discarded where Qwen's output diverged from ground truth (severity: minor, trajectory: improving)
- ForagerAgent data is unfiltered by success — the model learns from both successful and failed bug-fix trajectories (severity: minor, trajectory: stable)

## Bottlenecks Addressed and Created

**Addresses:**
- The foundational bottleneck that [[themes/code_and_software_ai|LLMs trained on static code lack grounded understanding of execution semantics]] — CWM is the first open-weights model to attack this at mid-training scale with a purpose-built dataset

**Exposes:**
- No principled methodology yet exists for integrating execution trace prediction into natural language reasoning chains — the *mechanism* by which trace grounding should improve reasoning or code generation remains formally unspecified
- Building executable repository images at scale is an unsolved engineering challenge that currently caps the size and diversity of world-model training data
