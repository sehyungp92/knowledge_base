### Motivation & Prior Limitations
- LLM agents that produce actions as JSON or text in pre-defined formats are bottlenecked by constrained action spaces and inflexible composition, making it difficult to tackle complex real-world tasks that require chaining multiple tools or dynamic adaptation.
  - Text and JSON action formats require each tool to be hand-crafted for a specific task; there is no natural mechanism for composing tools (e.g., passing output of one tool as input to another) or expressing control flow such as loops and conditionals without bespoke engineering.
  - Existing code-generating agents (e.g., for robot control) rely on pre-specified control primitives and hand-engineered prompts, and critically cannot dynamically revise or emit new actions based on new environmental feedback mid-task.
- There was no benchmark designed to evaluate agents on complex tasks requiring multi-tool composition in multi-turn interactions, leaving this performance gap unmeasured against alternative action formats.

---

### Proposed Approach
- CodeAct is a general-purpose framework that unifies all LLM agent actions into a single executable Python code action space, integrated with a live Python interpreter that returns execution results (including error messages) as observations in a multi-turn interaction loop.
  - Unlike JSON/text approaches, Python's native control flow (for-loops, if-statements) and data flow (variables, function composition) let the agent express arbitrarily complex multi-tool operations within a single action, rather than requiring one action per tool invocation.
  - Because LLMs are already extensively pre-trained on code, CodeAct requires no specialized format data; the model can directly import and use any PyPI package, and automated error tracebacks serve as a built-in self-debugging feedback signal without additional human engineering.
- To realize this for open-source models, the authors construct CodeActInstruct, a 7k-instance instruction-tuning dataset of multi-turn agent-environment trajectories spanning information seeking (HotpotQA), software package use (MATH, APPS), external memory (WikiTableQuestions), and robot planning (ALFWorld), generated by GPT-3.5, Claude, and GPT-4 with quality filtering to retain trajectories where the model self-corrects errors across turns.
  - CodeActAgent is then trained by fine-tuning Llama-2 7B and Mistral 7B on a mixture of CodeActInstruct and general conversation data (OpenOrca, ShareGPT, CapyBara), preserving both agent capability and general language ability.

---

### Results & Capabilities
- On the new M3ToolEval benchmark (82 human-curated tasks requiring multi-turn, multi-tool composition), CodeAct achieves a higher task success rate than JSON or text actions for 12 of 17 evaluated LLMs, with GPT-4-1106-preview reaching 74.4% success — a 20.7 percentage-point absolute improvement over the next-best format (text at 53.7%) — while requiring 2.1 fewer interaction turns on average.
  - CodeAct also requires fewer total actions across the board: it achieves the lowest average turn count in 12 of 17 models, confirming that tool composition within a single code action reduces back-and-forth.
- On atomic API-Bank tool calling (where CodeAct's control/data flow advantage is ablated), CodeAct still achieves comparable or better performance than text and JSON for most LLMs, with gains most pronounced for open-source models (JSON is particularly weak for open-source LLMs, never winning outright in this condition).
  - This suggests that code is a more natural action format for models trained on abundant code data, even for simple single-tool invocations.
- CodeActAgent (Mistral 7B) achieves 57.4% on in-domain MINT and 32.4% on out-of-domain MINT, outperforming all evaluated open-source LLMs of similar size and reaching performance comparable to 70B models on M3ToolEval (12.2% vs. best open-source 13.4%).
  - CodeActAgent also generalizes to text-action out-of-domain tasks (MiniWob++, SciWorld) without explicit text-action training, matching AgentLM-7B which was explicitly tuned for text actions.
  - General capability benchmarks (MMLU, HumanEval, GSM8K, MT-Bench) are maintained or improved for CodeActAgent relative to the base instruction-tuned models, demonstrating that agent-oriented fine-tuning with the mixed data recipe does not degrade general LLM ability.
- Qualitatively, CodeActAgent (Mistral 7B) can autonomously execute sophisticated multi-step workflows — loading datasets with Pandas, training sklearn regression models, visualizing results with Matplotlib, and self-debugging multiple rounds of AttributeError and ValueError — all without any in-context demonstrations.

### Implications
- Treating executable Python code as the universal action interface for LLM agents effectively removes the bottleneck of hand-crafted tool libraries: any Python package on PyPI becomes immediately usable, dramatically lowering the engineering overhead for deploying agents to new domains.
- The finding that CodeAct benefits scale with model capability (performance gains widen as LLM quality increases, especially for closed-source frontier models) suggests that as base models improve, the advantage of code-structured action spaces will compound — making this a durable architectural choice rather than a narrow benchmark optimization.
- CodeActInstruct demonstrates a reproducible distillation recipe for agent capabilities: synthetic trajectory generation from stronger models, filtered for self-correction behavior, combined with general conversation data, transfers meaningful agent skill without catastrophic forgetting — a template applicable beyond CodeAct specifically.

## Key Claims

1. LLM agents using JSON or text action formats suffer from constrained action spaces and restricted flexibility, including inability to compose multiple tools in a single action.
2. CodeAct achieves up to 20% higher success rate compared to JSON and text action alternatives across 17 evaluated LLMs.
3. CodeAct reduces the number of required interaction turns by up to 30% compared to JSON and text alternatives on complex multi-tool tasks.
4. CodeAct achieves higher task success rates than alternative action formats in 12 out of 17 evaluated LLMs on M3ToolEval.
5. Performance gains of CodeAct over text and JSON widen as the underlying LLM's capabilities increase.

7. JSON action format is consistently weaker than text and CodeAct for open-source LLMs, but achieves decent performance with closed-source LLMs, suggesting targeted fine-tuning by closed-source model pr
8. CodeAct is more natural for LLMs to use for tool-calling because LLMs have extensive exposure to code data during pre-training.
9. CodeAct enables LLMs to leverage existing Python software packages as tools, eliminating the need for hand-crafted task-specific toolsets.
10. Automated error feedback (e.g., tracebacks) from Python interpreters enables LLM agents to self-debug without requiring human-engineered feedback mechanisms.

## Capabilities

- LLM agents using executable Python code (CodeAct) as actions can perform complex multi-tool tasks requiring tool composition, looping, and control flow — achieving up to 74.4% success rate on complex multi-tool benchmarks with GPT-4 and up to 20% absolute improvement over JSON/text baselines
- LLM agents integrated with a Python interpreter can autonomously self-debug by reading error tracebacks and iteratively revising generated code in multi-turn interactions without in-context demonstrations
- 7B fine-tuned open-source agents (CodeActAgent on Mistral-7B) can use the full PyPI ecosystem (Pandas, Scikit-Learn, Matplotlib) for sophisticated tasks including ML model training and data visualization without bespoke tool definitions
- Instruction-tuned agents trained solely on CodeAct code-action trajectories can generalise to text-action format out-of-domain tasks, achieving comparable performance to models explicitly trained on text actions
- LLM agents can leverage the full PyPI ecosystem directly as tools — any Python package becomes an available action without hand-crafted API wrappers, tool schemas, or per-task human engineering

## Limitations

- Open-source LLMs (7B–70B) perform drastically worse than closed-source models on complex zero-shot multi-tool agentic tasks — best open-source achieves only 13.4% vs 74.4% for GPT-4 on M3ToolEval, a greater than 5× performance gap
- CodeAct performance gains over JSON/text are model-capability-dependent — weaker models see little to no benefit, with the advantage widening as base model capability increases, making the framework unusable for low-capability models
- CodeActAgent (LLaMA-2 7B variant) achieves 0.0% on out-of-domain M3ToolEval — complete failure on complex zero-shot multi-tool tasks, indicating a hard capability floor below which code-action frameworks provide no benefit
- All evaluations impose a hard 10-turn interaction limit, artificially capping agent performance and leaving unknown how accuracy degrades for tasks genuinely requiring more iterations — success rates may be systematically underestimated
- Self-debugging through automated error feedback can require many rounds without convergence — Figure 3 shows 3+ rounds of debugging for a simple matplotlib rotation task, revealing fragility when LLM API knowledge is stale or incorrect
- M3ToolEval benchmark contains only 82 human-curated instances — statistical significance of reported 20% improvements is weak, and benchmark may not represent the full distribution of real-world multi-tool tasks
- CodeActInstruct training data is limited to 7k trajectories — extremely small relative to general instruction tuning corpora (69k general conversation examples used alongside it), revealing how data-scarce high-quality agent trajectory generation remains
- Agent instruction tuning creates a general capability trade-off — CodeActAgent (Mistral) shows MMLU degradation despite mixing with general conversation data, suggesting agent specialisation partially compromises knowledge breadth
- Security, sandboxing, and real-world deployment risks of executing LLM-generated arbitrary Python code are entirely absent from the analysis — no discussion of code injection, resource exhaustion, data exfiltration, or isolation mechanisms
- Closed-source model JSON API calling matches or exceeds CodeAct for atomic tool use — the CodeAct advantage is significantly reduced for simple single-tool scenarios, indicating the framework's value is conditional on task complexity

## Bottlenecks

- Open-source LLM zero-shot instruction following is insufficient for complex multi-tool agentic tasks — models require demonstrations or specialised fine-tuning to approach closed-source zero-shot performance, blocking practical deployment without per-task prompt engineering
- Pre-defined action format constraints (JSON schemas, text templates) fundamentally limit agent action space to manually enumerated tools, blocking flexible multi-tool composition, iteration, and access to the general software ecosystem
- High-quality multi-turn agent interaction trajectory generation is expensive and data-scarce — the gap between available agent training data (thousands of trajectories) and general instruction tuning data (millions of examples) limits agent specialisation at scale

## Breakthroughs

- Executable Python code as a unified LLM agent action space (CodeAct) — replacing pre-defined JSON/text tool schemas with native Python execution, enabling arbitrary control flow, PyPI ecosystem access, automated error feedback, and self-debugging in a single general-purpose framework
